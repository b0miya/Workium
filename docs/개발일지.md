### 1단계 — 프로젝트 구성 및 Firebase 연동

**작업 내용**
- 4개 파일로 단일 페이지 앱 구현
  - `index.html` — 전체 UI 마크업
  - `app.js` — 로직 전체 (Firebase CRUD, 렌더링, 모달)
  - `style.css` — 스타일
  - `firebase-config.js` — Firebase 연결 설정
- Firebase 프로젝트 생성 및 Firestore 연결 완료
- 4개 Firestore 컬렉션 설계

| 컬렉션 | 필드 |
|---|---|
| `members` | id, name, role, email, color, createdAt |
| `tasks` | id, memberId, title, date, memo, createdAt |
| `sites` | id, name, url, desc, createdAt |
| `config/notice` | text |

**결정 사항**
- Firebase Auth(로그인) 없이 운영 → sessionStorage로 접속자 식별
- 역할 기반 권한: `role === '부장'`이면 편집 권한 부여

### 2단계 — 프로젝트명 변경 (jobpro → Workium)

**Q. 프로젝트 폴더 이름이 jobpro에서 Workium으로 바뀜**

코드 내 `jobpro` 참조 확인 결과, `sessionStorage` 키 `jp_user` 발견.

**변경 내용**
```js
// 변경 전
sessionStorage.getItem('jp_user')
sessionStorage.setItem('jp_user', id)
sessionStorage.removeItem('jp_user')

// 변경 후
sessionStorage.getItem('workium_user')
sessionStorage.setItem('workium_user', id)
sessionStorage.removeItem('workium_user')
```

> 기존에 로그인된 사용자는 변경 후 다시 선택 필요

### 4단계 — Firebase 보안 규칙 확인

**Q. 이 주소를 부서원에게 공유하면 되니?**

Firebase Firestore 보안 규칙 확인 결과:

```
allow read, write: if true;  ← 누구나 접근 가능한 상태
```

**결론**
- Firebase Auth 없이 운영하는 구조상 완벽한 차단은 어려움
- 부서 내부용 소규모 도구이므로 현재 상태로 운영해도 실용상 문제없음
- 추후 보안 강화 시 Firebase Authentication 추가 검토

### 5단계 — 프로젝트 파일 분석

**파일별 역할 요약**

**`firebase-config.js`**
- Firebase 프로젝트 연결 설정값 보관

**`app.js`** — 핵심 로직
- Firebase 실시간 리스너 4개 (`onSnapshot`)
- `_checkReady()`: 4개 컬렉션 모두 로드 완료 후 UI 렌더링
- `isBoss()`: 현재 접속자의 role이 '부장'인지 확인
- 뷰 전환: `selectView('all' | 'member' | 'manage')`

**`index.html`** — UI 구조
- 좌측 사이드바: 부서원 목록, 접속자 표시
- 우측 메인: 전체보기 / 개인뷰 / 관리 페이지
- 모달 4개: 업무, 부서원, 접속자 선택, 사이트

**`style.css`**
- CSS Custom Properties 기반 디자인 토큰
- 2컬럼 레이아웃 (사이드바 230px + 메인)
- 모바일 반응형 (760px 이하 사이드바 축소)

### 7단계 — 권한 버그 수정 (1차)

**Q. 부장이 아니어도 부서원 관리와 부서원 추가 메뉴가 활성화 됨**

**원인 분석**
1. HTML에서 `manage-btn`이 기본으로 표시되는 상태로 시작
2. 멤버 데이터 업데이트 시 `renderCurrentUser()`가 호출되지 않아 버튼 상태 미갱신

**수정 내용**

`index.html` — 기본값 숨김 처리
```html
<!-- 변경 전 -->
<button class="manage-btn" onclick="openMembersPage()">

<!-- 변경 후 -->
<button class="manage-btn" style="display:none" onclick="openMembersPage()">
```

`app.js` — 멤버 데이터 변경 시 권한 재평가
```js
db.collection('members').onSnapshot(snap => {
  members = ...
  renderSidebar();
  renderCurrentUser();  // 추가
  _refreshView();
});
```

### 8단계 — 권한 버그 수정 (2차) — 초기 설정 문제

**Q. 부장에게 편집 권한이 있으나 부서원 관리의 부서원 추가 기능이 없어짐**

**원인 분석 (닭-달걀 문제)**
- 부서원이 0명 → 로그인 불가 → 부장 지정 불가 → 부서원 관리 접근 불가
- 첫 번째 부서원(부장)을 추가할 방법이 없는 상태

**수정 내용**
```js
// 부서원 관리 버튼: 부장이거나 멤버가 없을 때 표시
document.querySelector('.manage-btn').style.display
  = (isBoss() || members.length === 0) ? 'flex' : 'none';

// 관리 뷰 진입: 부장이거나 멤버가 없을 때 허용
if (!isBoss() && members.length > 0) {
  showToast('부장만 접근할 수 있습니다.');
  return;
}
```

### 9단계 — 권한 버그 수정 (3차)

**Q. 부장이 아닌 경우에는 부서원 추가 항목도 보이지 않아야 해**

**수정 내용**
```js
// 변경 전: 관리 뷰 진입 시 항상 표시
document.getElementById('addMemberBtn').style.display = 'inline-flex';

// 변경 후: 부장만 표시
document.getElementById('addMemberBtn').style.display
  = isBoss() ? 'inline-flex' : 'none';
```

### 10단계 — 권한 버그 수정 (4차)

**Q. 초기 진입에서 부장으로 들어갔다가 다른 부원을 클릭해서 부원으로 바꿔 들어가면 부서원 추가 항목이 보여**

**원인 분석**
- 접속자 전환 시 `renderCurrentUser()`는 호출되지만 `addMemberBtn` 상태를 갱신하지 않음

**수정 내용**
```js

  = isBoss() ? 'inline-flex' : 'none';
```

## 동작 메커니즘 요약

### 앱 초기화 흐름
```
페이지 로드
  → 로딩 스피너 표시
  → Firebase 실시간 리스너 4개 동시 연결
  → 4개 모두 로드 완료 → _checkReady() 실행
  → UI 렌더링 완료 → 로딩 스피너 숨김
  → 로그인 유저 없으면 접속자 선택 모달 표시
```

### 실시간 데이터 동기화
```
Firestore 데이터 변경 감지 (onSnapshot)
  → members 변경 → 사이드바 + 접속자 + 뷰 갱신
  → tasks 변경  → 현재 뷰 갱신
  → sites 변경  → 사이트 패널 갱신
  → notice 변경 → 공지 패널 갱신
```

### 권한 체계
```
접속자 선택 (sessionStorage 저장)
  → isBoss() 평가
      부장  → 부서원 관리 / 공지 편집 / 사이트 관리 가능
      부서원 → 전체 보기 / 본인 업무 입력만 가능
      미선택 → 읽기 전용
      멤버 0명 → 초기 설정을 위해 관리 접근 허용
```
